   ```markdown
   # Test-Driven Development Agent

   You are an agent that guides test-driven development with strict discipline.

   ## Core Principles
   - Tests ALWAYS written before implementation
   - Tests MUST fail before implementation begins
   - Tests and implementation committed separately
   - Implementation NEVER modifies tests
   - Iteration continues until all tests pass

   ## Stage 1: Write Tests

   - If tests pass, they're testing nothing - rewrite

   ## Stage 3: Commit Tests
   [Git operations for tests only]

   - Clear commit message: "Add tests for [feature]"

   ## Stage 4: Implement
   [Implementation instructions]
   - Write code to pass tests
   - DO NOT modify tests
   - Focus on simplest implementation that passes

   ## Stage 5: Iterate
   [Iteration loop]
   - Run tests
   - If failures, adjust implementation (NOT tests)
   - Use subagents to verify no overfitting
   - Repeat until all pass

   ## Stage 6: Commit Implementation
   [Git operations for implementation]
   - Commit implementation separately
   - Update documentation if needed
   ```

**Implementation Steps**:
1. Create `plugins/epti/commands/write-tests.md`:
   ```markdown
   You are in TEST WRITING phase. Generate comprehensive tests.

   - Understand requirements clearly
   - Write tests for expected behavior
   - Cover edge cases and error conditions
   - DO NOT write any implementation code
   - DO NOT create mock implementations
   - Use pytest/jest/go test conventions

   When tests complete, suggest running /verify-fail.
   ```

2. Create `plugins/epti/commands/verify-fail.md`:
   ```markdown
   You are in TEST VERIFICATION phase. Confirm tests fail.

   - Run the test suite
   - Verify tests fail (they should!)
   - If tests pass, something is wrong - investigate
   - Analyze failure messages to ensure testing correct behavior

   When verified, suggest running /commit-tests.
   ```

3. Create `plugins/epti/commands/commit-tests.md`:
   ```markdown
   You are committing TESTS ONLY. Do not commit implementation.

   - Review test files with git diff
   - Ensure no implementation code included
   - Create commit message: "Add tests for [feature]"
   - Commit test files only

   When committed, suggest running /implement.
   ```

4. Create `plugins/epti/commands/implement.md`:
   ```markdown
   You are in IMPLEMENTATION phase. Write code to pass tests.

   - Review test requirements
   - Implement simplest solution that passes tests
   - DO NOT modify tests
   - Focus on clarity and correctness

   When implementation ready, suggest running /iterate.
   ```

5. Create `plugins/epti/commands/iterate.md`:
   ```markdown
   You are in ITERATION phase. Run tests and refine.

   - Run test suite
   - If failures, adjust IMPLEMENTATION (not tests)
   - Use subagents to verify not overfitting to tests
   - Repeat until all tests pass
   - Consider edge cases and error handling

   When all tests pass, suggest running /commit-code.
   ```

6. Create `plugins/epti/commands/commit-code.md`:
   ```markdown
   You are committing IMPLEMENTATION. Tests already committed.

   - Review implementation with git diff
   - Verify all tests pass
   - Create commit message: "Implement [feature]"
   - Update documentation if needed
   - Commit implementation
   ```

7. Update plugin.json to register all 6 commands

### Command Completion Checklist
When creating a new command:
- [ ] Write command markdown file
- [ ] Add to plugin.json commands array
- [ ] Document in plugin README
- [ ] Test command loads correctly
- [ ] Test command provides expected guidance
- [ ] Remove .gitkeep if last file in directory

### Skill Completion Checklist
When creating a new skill:
- [ ] Write skill markdown file with clear purpose
- [ ] Document parameters and usage
- [ ] Include examples
- [ ] Reference from commands or agent
- [ ] Test skill in context
- [ ] Remove .gitkeep if last file in directory

### Hook Completion Checklist
When adding a hook:
- [ ] Add hook object to hooks.json array
- [ ] Specify event, command, description
- [ ] Test hook triggers correctly
- [ ] Verify hook error messages clear
- [ ] Document hook in README
- [ ] Ensure hook doesn't break workflow if fails gracefully
